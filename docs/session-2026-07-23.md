# Session du 23 juillet 2026 — Base de données : Phase 1 (fondation) + Phase 2 (import)

Grosse session : mise en place de la vraie base de données (fin du localStorage
comme stockage principal, à terme) et de l'assistant d'import de l'ancienne base
STCA M. Plus deux finitions visuelles validées.

## Contexte / décision
L'utilisateur veut pouvoir **importer l'ancienne base STCA** (338 075
enregistrements) — impossible avec le localStorage (~5 Mo). But final =
**client/serveur**. Décision validée : **SQLite + Prisma maintenant**, passage à
PostgreSQL/serveur plus tard (changement de provider, pas de réécriture).
Détails : `docs/base-de-donnees.md`.

## Phase 1 — Fondation base de données (VALIDÉE)
- `prisma/schema.prisma` : **9 modèles fidèles à STCA M** (SQLite). Décisions
  validées : archives = UNE table (`dateArchivage` nul = actif), mots de passe
  en clair pour l'instant (hachage Phase 3), portée = tout STCA M (import sans
  perte). Points fidèles : couple `(numImmatriculation, codeTransit)` UNIQUE,
  `vin` et `numRef` uniques, `categorieRang`/`codeTransit` en réf. souples.
  Ajouts TCIT : `couleur` (Destination), détail des primes (TarifAssurance).
- Prisma 6.19.3 (stable ; Prisma 7 écarté — config par adapters trop récente).
  Migration `init_stca_m` → 9 tables. Base `prisma/stca.db` (ignorée par git).
- Test end-to-end du client Prisma OK (écriture, lecture, refus de doublon
  immat+frontière).

## Phase 2 — Assistant d'import de l'ancienne base (VALIDÉE)
Trois briques nouvelles, toutes testées en Electron réel :
1. **Prisma dans le process principal** — `src/main/db.ts` (client singleton,
   chemin absolu de la base, IPC `db:counts`). L'intégration Prisma+Electron —
   le point le plus risqué — fonctionne (sonde appelée depuis le renderer).
2. **Accès disque + aperçu CSV** — `src/main/import.ts` : dialogue natif
   (`import:pickFile`), aperçu par lecture d'un extrait (`import:preview`,
   détection du délimiteur, en-têtes + 20 lignes), import par lots
   (`import:run`).
3. **Import + rapport** — coercition des types (dates AAAAMMJJ / JJ/MM/AAAA,
   montants, booléens), dédoublonnage sur les DEUX contraintes uniques (couple
   immat+frontière ET châssis), validation des champs requis avec n° de ligne,
   createMany par lots de 500 + repli ligne par ligne, progression émise au
   renderer. csv-parse 7.
- **Interface** `src/renderer/src/pages/ImportAssistant.tsx` : 3 étapes
  (Fichier → Correspondance → Import). Correspondance des colonnes **pré-remplie
  automatiquement** (alias des noms STCA), aperçu, rapport
  (lues/importées/doublons/erreurs + détail). Testée end-to-end via un CSV
  d'exemple (8 lignes → 4 importées, 2 doublons, 2 erreurs) — contournement
  temporaire du dialogue natif pendant le test, retiré ensuite.
- **Fenêtre `outils.exporter` transformée** en outil de données à onglets
  (`ExportPage.tsx`) : Import ancienne base (actif) · Sauvegarde · Restauration ·
  Export rapport (ces 3 = « à venir Phase 3 »). Titre « Données — Sauvegarde,
  Restauration & Import », 800×660, **garde admin** (`FENETRES_ADMIN`).
- IPC exposés dans `preload/index.ts` + `@api/electron` (dbCounts, importPickFile,
  importPreview, importRun, onImportProgress).

⚠️ Les données importées ne seront visibles dans l'app qu'après la **Phase 3**
(bascule des stores vers la base). C'est indiqué dans l'assistant.
